yocto 빌드시 특정패키지에서 internal compiler error 발생관련 (boost 빌드에러)

yocto 빌드시 특정 패키지에서 알수없는 애러가 발생하는경우가 발생하였습니다.

일명 internal compiler error

문제증상

yocto 를 사용하여 모든 패키지를 풀빌드할때 주로 발생하는데, 발생하는 패키지는 랜덤인것 같습니다.

저같은경우 주로 boost 패키지에서 계속적으로 발생하더군요. 에러로그를 분석하려고 해도, 한정적인 정보도 그렇고 왜발생하는지 아무리 봐도 알수가 없었습니다.

  • 관련한 이슈는 저말고도 다른분들도 간혹격는것 같습니다.
    • 이슈 메일링 링크 : https://www.yoctoproject.org/pipermail/yocto/2019-January/043902.html

발생원인

사실 발생원인에 대해서는 추측만합니다. 관련하여 다른 개발자 지인과 이야기를 해보니… 그냥 host pc 가 문제인것 같습니다.

  • 특정 cpu 에러? os 와의 호환성?
    • 라이젠최근 cpu 에서 우분투 18.04 에서 발생하는것 같습니다.

문제해결방법

임시방편 조치들

OS 업데이트

지인의 경우 OS 업데이트를 하여 해결을 했다고 합니다. 즉 특정 cpu 와 os 간의 물리적인 호환성 문제로 보고 업데이트를 하였던것 것이었습니다.

하지만, os업데이트는 때때로 불가능할때도있습니다. ㅠㅠ

yocto version 에 따라서 특정 os를 지원하지 않기 때문입니다. yocto 에서 recommand 하지 않은 os 버젼을 사용하게되면 host compile 할때 에러가 발생합니다. ㅠㅠ

만약, os업데이트가 귀찮거나 힘드시면 아래의 다른 조치도 한번 고려해보세요.

yocto 다시 빌드 시도하기

원인도 잘 모르는데.. 희안한건 다시 bitbake 명령어를 통해 빌드를 하면 정상적으로 넘어갑니다.

그냥 bitbake 명령어가 안될때는… 에러가 나는 패키지를 bitbake -c do_cleanall <error package 를 해서 깨끗하게 지워주시고, 다시 빌드해봅니다. 이때 sstate-cache 폴더도 삭제해주면 확실하게 빌드가 넘어갑니다.

ㅠㅠ 한마디로 해결책이 엉망진창입니다. 이런해결은 그냥 임시방편.

cpu의 동시 작업 이슈인것으로 판단하여 조치

저같은경우 boost 패키지에서 계속적으로 다음과 같은 에러가 났습니다.

특이사항은 간혹 pc 가 엄청 느려질 만큼 job 들이 꼬여지는 일이 발생하였습니다.

일단, 특정 cpu + os 에서만 발생하고 코드상에는 전혀 문제가 없고, pc 가 물리적으로 느려지는것으로 볼때 컴파일러의 job 과 관련이 생각되어 다음과 같이 수정을 해봣습니다.

boost 패키지에 대한 job 수정

계속 말썽을 일으켰던 boost 의 레시피를 보면 다음과 같은 내용이 있습니다.

-/boost/boost.inc

BOOST_PARALLEL_MAKE 변수가 컴파일러 job 에 대한 변수인것 같습니다.

해당의 limit 을 limit=4 와 같이 숫자를 조정하여 job 갯수를 줄여줍니다. 즉 job 이문제가 되니, job 갯수를 확줄여버린거죠.

위의 조정으로 boost 에 대한 문제는 해결되었습니다!!

특정 패키지에서 internal compiler error 문제가 발생한다면, 해당 패키지의 compile job 옵션을 조정하셔서 문제가 생기지 않는 값을 찾는것도 좋을것 같습니다.

  • 참고
    • yocto 에서 빌드 쓰레드 관련해서 (-j 옵션) 한 변수들은 다음과 같습니다.
    • PARALLEL_MAKE, BB_NUMBER_THREADS 해당 변수와 함께 레시피에서 엮여있는 경우도 있으니 해당 변수도 같이 살펴보시면됩니다.

레시피 WORKDIR 에 대해서

일단 각 레시피를 빌드 하게 되면, 각각의 레시피별로 WORKDIR 폴더가 만들어집니다.

WORKDIR 폴더에 대한 대략적인 설명은 다음의 공식 홈페이지를 보면 나옵니다.

  • https://www.yoctoproject.org/docs/1.8/ref-manual/ref-manual.html

2.3.1. Task Failures
The log file for shell tasks is available in ${WORKDIR}/temp/log.do_taskname.pid. For example, the do_compile task for the QEMU minimal image for the x86 machine (qemux86) might be tmp/work/qemux86-poky-linux/core-image-minimal/1.0-r0/temp/log.do_compile.20830. To see what BitBake runs to generate that log, look at the corresponding run.do_taskname.pid file located in the same directory.

  • https://www.yoctoproject.org/docs/1.8/ref-manual/ref-manual.html#var-WORKDIR

The pathname of the work directory in which the OpenEmbedded build system builds a recipe. This directory is located within the TMPDIR directory structure and is specific to the recipe being built and the system for which it is being built.

The WORKDIR directory is defined as follows:

${TMPDIR}/work/${MULTIMACH_TARGET_SYS}/${PN}/${EXTENDPE}${PV}-${PR}

The actual directory depends on several things:

  • TMPDIR: The top-level build output directory
  • MULTIMACH_TARGET_SYS: The target system identifier
  • PN: The recipe name
  • EXTENDPE: The epoch - (if PE is not specified, which is usually the case for most - recipes, then EXTENDPE is blank)
  • PV: The recipe version
  • PR: The recipe revision

As an example, assume a Source Directory top-level folder name poky, a default Build Directory at poky/build, and a qemux86-poky-linux machine target system. Furthermore, suppose your recipe is named foo_1.3.0-r0.bb. In this case, the work directory the build system uses to build the package > would be as follows:

poky/build/tmp/work/qemux86-poky-linux/foo/1.3.0-r0

간단히 설명하면… tmp/work/qemux86-poky-linux/core-image-minimal/1.0-r0 와 같이, /tmp/work/아키텍쳐-프로젝트명/레시피이름/버젼명/ 의 경로가 기본 WORKDIR 입니다.

빌드한 패키지의 WORKDIR의 경로에 각종 정보가 있습니다.

  • 각종 빌드로그
  • git clone 한 외부 코드 / 내부 레시피에서 복사한 코드
  • 패치파일

에러가 났을때, 로그 확인하기

일단, 에러가 났다면 다음과 같이 -v 옵션을 켜서 빌드합니다.

빌드과정중에, 직관적으로 에러가 나는부분에 대해서 로그를 보시고 디버깅을 시작하시면됩니다. 또한, 특정 과정중에 에러가 났다면 WORKDIR 내의 temp 폴더 밑에 가면 로그들도 유용한 역을 합니다.

예를들어, busybox 를 빌드하다 do_install 과정중에 어러가 났다고 한다면, WORKDIR 내의 temp 폴더에 가면 다음과 같이 do_install 에 대한 로그들을 볼수있습니다.

  1. log.do_install 파일에 빌드과정중의 에러들이 기록되어있을겁니다. 해당 기록만봐서는 정확히 어느과정중에 에러가 났는지 알기 어렵습니다.
  2. log.do_install 의 내용을 run.do_install 와 함께 유추해봅니다. 해당 파일은 욕토 빌드과정중에 완성된 형태의 스크립트 입니다. 즉, 욕토과정중에 어떠한 변수들이 어떻게 들어갔는지 해당 파일을 열어보면 알수있습니다. (컴파일과정이라면, 컴파일러 옵션까지 모두 확인가능합니다.) log.do_install 로그는 완성된 형태의 스크립트 실행도중에 에러를 기준으로 남기게 됩니다.
